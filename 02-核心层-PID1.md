# 核心层分析 — PID 1（systemd 核心）

## 一、概述

PID 1 是 systemd 的核心，负责：
- 系统初始化与关机
- 管理所有系统单元（Unit）的生命周期
- 依赖关系解析与任务调度
- 进程执行与监控
- D-Bus 接口对外暴露系统管理 API

核心源码位于 `src/core/`，关键文件：

| 文件 | 职责 |
|------|------|
| main.c | PID 1 入口，模式判断，初始化流程 |
| manager.c / manager.h | 全局管理器，事件循环，队列管理 |
| unit.c / unit.h | 单元基类，依赖管理，状态跟踪 |
| service.c / service.h | 服务类型单元实现 |
| job.c / job.h | 任务（启动/停止操作）及事务管理 |
| execute.c / execute.h | 进程执行（fork/exec），沙箱配置 |
| load-fragment.c | 单元文件解析（.service/.socket 等） |
| transaction.c | 任务事务处理（依赖解析） |

## 二、启动流程

### 2.1 PID 1 入口 — main.c

```
main()
  ├── 判断运行模式（system/user）
  ├── 解析内核命令行参数
  ├── 设置日志级别
  ├── console_setup() — 配置控制台
  ├── manager_new() — 创建 Manager 实例
  ├── manager_startup() — 加载单元、恢复状态
  ├── manager_loop() — 进入 sd_event 事件循环
  └── manager_free() — 清理退出
```

关键行为：
- **重新执行（reexec）**：支持 `daemon-reexec` 命令，PID 1 可以在不重启的情况下更新自身
- **序列化/反序列化**：通过 `manager_serialize()` / `manager_deserialize()` 实现跨重新执行的状态持久化
- **双模式**：同一代码支持 system 模式（PID=1）和 user 模式（用户会话管理）

### 2.2 Manager 结构体

`Manager` 是 PID 1 的核心数据结构（定义于 `manager.h:143-420`），是整个系统的"大脑"：

```c
struct Manager {
    // === 单元管理 ===
    Hashmap *units;                    // 所有单元的哈希表（按名称索引）
    Hashmap *units_by_invocation_id;   // 按调用 ID 索引
    LIST_HEAD(Unit, units_by_type[_UNIT_TYPE_MAX]); // 按类型分组的链表

    // === 任务管理 ===
    Hashmap *jobs;                     // 所有活跃任务
    LIST_HEAD(Job, run_queue);         // 待执行任务队列

    // === 事件循环 ===
    sd_event *event;                   // 核心事件循环

    // === D-Bus ===
    sd_bus *api_bus;                   // API 总线（外部通信）
    sd_bus *system_bus;                // 系统总线

    // === 通知队列 ===
    LIST_HEAD(Unit, load_queue);       // 待加载单元队列
    LIST_HEAD(Unit, dbus_unit_queue);  // 待发 D-Bus 信号的单元
    LIST_HEAD(Unit, cleanup_queue);    // 待清理队列
    LIST_HEAD(Unit, gc_unit_queue);    // 垃圾回收队列
    LIST_HEAD(Unit, start_when_upheld_queue); // 依赖启动队列
    LIST_HEAD(Unit, stop_when_unneeded_queue); // 不再需要时停止队列

    // === 进程监控 ===
    Hashmap *watch_pids;               // 监控的子进程 PID

    // === cgroup 管理 ===
    char *cgroup_root;                 // cgroup 根路径
    CGroupMask cgroup_supported;       // 支持的 cgroup 控制器

    // === 默认配置 ===
    ExecContext default_exec_context;  // 默认执行上下文
    KillContext default_kill_context;  // 默认 kill 策略
    CGroupContext default_cgroup_context; // 默认 cgroup 配置

    // === 时间戳 ===
    dual_timestamp timestamps[_MANAGER_TIMESTAMP_MAX]; // 启动时间等

    // === 状态 ===
    ManagerObjective objective;        // 当前目标（运行/重启/关机等）
    Set *startup_units;                // 正在启动的单元集合
    Set *failed_units;                 // 失败单元集合
    ...
};
```

**核心设计要点**：
1. **多队列架构**：通过不同的链表队列（load_queue、run_queue、dbus_unit_queue 等）分阶段处理工作
2. **哈希表索引**：units 和 jobs 使用 Hashmap 快速查找
3. **事件驱动**：所有 I/O、定时器、信号都通过 sd_event 统一处理

## 三、Unit 单元系统

### 3.1 Unit 基类

`Unit` 是所有单元类型的基类（`unit.h:130-430`）：

```c
struct Unit {
    Manager *manager;               // 指向管理器
    UnitType type;                  // 单元类型
    char *id;                       // 主名称（如 sshd.service）
    Set *aliases;                   // 别名集合

    // === 依赖关系 ===
    Hashmap *dependencies;          // 依赖关系图（UnitDependency → Unit set）

    // === 状态 ===
    UnitLoadState load_state;       // 加载状态（stub/loaded/error/...）
    UnitActiveState active_state;   // 活跃状态（active/inactive/...）

    // === 条件检查 ===
    LIST_HEAD(Condition, conditions);     // 启动条件
    LIST_HEAD(Condition, asserts);        // 启动断言

    // === cgroup ===
    CGroupContext cgroup_context;   // cgroup 配置
    char *cgroup_path;              // cgroup 路径
    CGroupMask cgroup_realized_mask; // 已实现的 cgroup 控制器

    // === 执行上下文 ===
    // 由子类（如 Service）持有

    // === GC 与引用计数 ===
    unsigned gc_marker;             // 垃圾回收标记
    LIST_FIELDS(Unit, gc_unit_queue); // GC 队列链接

    // === 队列链接 ===
    LIST_FIELDS(Unit, load_queue);
    LIST_FIELDS(Unit, dbus_unit_queue);
    LIST_FIELDS(Unit, cleanup_queue);
    ...
};
```

### 3.2 单元类型

systemd 支持 11 种单元类型，每种通过 `UnitVTable` 提供多态实现：

| 类型 | 说明 | 实现文件 |
|------|------|---------|
| service | 进程服务 | service.c |
| socket | 套接字（socket 激活） | socket.c |
| target | 目标（同步点） | target.c |
| device | 设备 | device.c |
| mount | 挂载点 | mount.c |
| automount | 自动挂载 | automount.c |
| swap | 交换分区 | swap.c |
| timer | 定时器 | timer.c |
| path | 路径监控 | path.c |
| slice | 资源切片（cgroup） | slice.c |
| scope | 外部进程作用域 | scope.c |

### 3.3 依赖关系

Unit 之间的依赖通过 `dependencies` 哈希表管理，支持多种依赖类型：

| 依赖类型 | 含义 |
|---------|------|
| Requires | 强依赖（依赖失败则自身也失败） |
| Wants | 弱依赖（依赖失败不影响自身） |
| After/Before | 启动顺序（不创建依赖关系，仅排序） |
| BindsTo | 绑定（比 Requires 更强，依赖停止则自身也停止） |
| Conflicts | 冲突（互斥） |
| Requisite | 前提条件（依赖必须已经处于活跃状态） |
| PartOf | 部分关系（随依赖一起重启/停止） |
| ConsistsOf | 组成关系 |

## 四、Job 任务系统

### 4.1 Job 结构体

`Job` 表示一个待执行的操作（`job.h:116-164`）：

```c
struct Job {
    Manager *manager;          // 管理器指针
    Unit *unit;                // 目标单元
    JobType type;              // 操作类型
    JobState state;            // 当前状态

    // === 事务图 ===
    LIST_HEAD(JobDependency, subject_list);  // 依赖此 Job 的其他 Job
    LIST_HEAD(JobDependency, object_list);   // 此 Job 依赖的其他 Job

    // === 执行队列 ===
    LIST_FIELDS(Job, run_queue);
    LIST_FIELDS(Job, dbus_queue);

    // === 超时 ===
    sd_event_source *timer_event_source;
    usec_t begin_usec;
    usec_t begin_running_usec;
    ...
};
```

### 4.2 Job 类型与状态

```
JobType: START, VERIFY_ACTIVE, STOP, RELOAD, RESTART, NOP
JobState: WAITING → RUNNING → (完成/失败)
```

### 4.3 事务处理

依赖解析通过 **事务（Transaction）** 机制完成：

```
用户请求 "start sshd.service"
  ↓
transaction_add_job() — 添加 START job
  ↓
遍历依赖关系，递归添加所需 jobs
  ↓
transaction_verify_order() — 检测循环依赖
  ↓
transaction_minimize() — 去除冗余 jobs
  ↓
transaction_apply() — 将 jobs 安装到 Manager 的 run_queue
  ↓
manager_run_jobs() — 按序执行
```

## 五、Execute 执行系统

### 5.1 ExecContext — 静态配置

`ExecContext` 保存从单元文件解析的执行配置（`execute.h:176-420`），涵盖：

| 分类 | 配置项示例 |
|------|----------|
| 用户/组 | User=, Group=, SupplementaryGroups= |
| 工作目录 | WorkingDirectory=, RootDirectory= |
| 环境变量 | Environment=, EnvironmentFile= |
| 资源限制 | LimitCPU=, LimitFSIZE=, LimitNOFILE= |
| 沙箱 | PrivateTmp=, ProtectSystem=, ProtectHome= |
| 命名空间 | PrivateNetwork=, PrivateUsers=, PrivateMounts= |
| 安全 | SELinuxContext=, AppArmorProfile=, SeccompFilter= |
| 能力 | CapabilityBoundingSet=, AmbientCapabilities= |
| 系统调用过滤 | SystemCallFilter=, SystemCallArchitectures= |
| 日志 | StandardOutput=, StandardError=, SyslogIdentifier= |
| 凭据 | LoadCredential=, SetCredential= |

### 5.2 ExecParameters — 运行时参数

`ExecParameters` 是每次执行时的动态参数，包括：
- 打开的文件描述符
- cgroup 路径
- 运行时目录
- 确认信息

### 5.3 执行流程

```
exec_spawn()
  ├── fork() — 创建子进程
  └── 子进程中:
      ├── exec_child() — 执行环境设置
      │   ├── 设置 cgroup
      │   ├── 设置命名空间（mount/network/pid/user）
      │   ├── 应用资源限制（rlimit）
      │   ├── 应用 seccomp 过滤器
      │   ├── 设置 SELinux/AppArmor 上下文
      │   ├── 切换用户/组
      │   ├── 设置环境变量
      │   ├── chroot/pivot_root（如果配置了）
      │   └── execve() — 执行目标程序
      └── 父进程：监控子进程退出
```

## 六、核心事件循环

Manager 使用 `sd_event` 事件循环处理所有异步事件：

| 事件源 | 处理内容 |
|--------|---------|
| signalfd | SIGCHLD（子进程退出）、SIGTERM/SIGINT（关机） |
| timerfd | 各种超时（任务超时、看门狗等） |
| D-Bus fd | API 请求（systemctl 等工具的命令） |
| inotify | 单元文件变更监控 |
| notify socket | sd_notify() 通知（服务就绪、状态更新等） |
| cgroup events | cgroup 空通知（所有进程退出） |

### 事件循环主循环

```
manager_loop() {
    while (m->objective == MANAGER_OK) {
        // 1. 处理加载队列中的单元
        manager_dispatch_load_queue(m);

        // 2. 处理待运行的 jobs
        manager_dispatch_run_queue(m);

        // 3. 发送 D-Bus 通知
        manager_dispatch_dbus_queue(m);

        // 4. 等待下一个事件
        sd_event_run(m->event, timeout);
    }
}
```

## 七、配置文件解析

单元文件（.service、.socket 等）通过 **gperf** 生成的查找表解析：

```
load-fragment-gperf.gperf → gperf 编译 → load-fragment-gperf.c
```

解析流程：
1. `unit_load()` 触发加载
2. `load_fragment()` 读取文件
3. gperf 查找表将配置键映射到解析函数
4. 解析函数填充 `ExecContext`、`KillContext`、`CGroupContext` 等结构体

## 八、关键调用链

### 启动一个服务

```
systemctl start foo.service
  → D-Bus 调用 → manager_start_unit()
    → manager_add_job() → transaction_add_job_and_dependencies()
      → transaction_activate() → job_install()
        → job 进入 run_queue
          → manager_dispatch_run_queue()
            → job_run_and_invalidate()
              → service_start()
                → service_spawn() → exec_spawn()
                  → fork() + exec_child() + execve()
```

### SIGCHLD 处理（子进程退出）

```
SIGCHLD 信号
  → manager_dispatch_sigchld()
    → waitid() 获取退出状态
      → unit_child_handler() → service_sigchld_event()
        → 更新服务状态机
          → job_finish_and_invalidate()
            → 触发依赖单元的后续操作
```
