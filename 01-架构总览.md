# systemd v250 架构总览

## 一、整体架构

systemd 是一个 Linux 系统和服务管理器，作为 PID 1 运行。整体架构分为以下层次：

```
┌─────────────────────────────────────────────────────────────────┐
│                      用户空间工具层                               │
│  systemctl  journalctl  loginctl  networkctl  resolvectl  ...  │
├─────────────────────────────────────────────────────────────────┤
│                      公共 API 层（libsystemd）                   │
│     sd-bus    sd-event    sd-journal    sd-device    sd-login   │
├─────────────────────────────────────────────────────────────────┤
│                      服务守护进程层                               │
│  journald   logind   networkd   resolved   timesyncd   udevd   │
│  homed   machined   portabled   oomd   coredump                │
├─────────────────────────────────────────────────────────────────┤
│                      核心层（PID 1）                             │
│  Manager → Unit → Job → Execute                                │
│  依赖解析 → 事务处理 → 状态机 → 进程执行                          │
├─────────────────────────────────────────────────────────────────┤
│                      基础库层                                    │
│  src/basic/（字符串、文件、进程、哈希表、时间等基础工具）             │
│  src/shared/（D-Bus辅助、安全模块、网络工具、启动配置等共享代码）     │
│  src/fundamental/（最底层，可在 EFI 环境中使用的基础定义）           │
├─────────────────────────────────────────────────────────────────┤
│                      Linux 内核接口                              │
│  cgroups v1/v2   namespaces   netlink   sysfs   procfs         │
│  uevent   inotify   epoll   signalfd   timerfd                 │
└─────────────────────────────────────────────────────────────────┘
```

## 二、核心设计模式

### 2.1 Manager + 事件循环模式

几乎所有 systemd 守护进程都遵循同一架构模式：

```c
// 典型守护进程 main() 流程
int main(int argc, char *argv[]) {
    Manager *m = NULL;
    manager_new(&m);           // 1. 创建管理器
    manager_connect_bus(m);    // 2. 连接 D-Bus
    manager_enumerate(m);      // 3. 枚举已有状态
    sd_event_loop(m->event);   // 4. 进入事件循环
    manager_free(m);           // 5. 清理退出
}
```

**关键发现**：systemd 中至少有 12 个不同的 `Manager` 结构体分布在不同子系统中，每个都持有自己的 `sd_event` 事件循环和 `sd_bus` D-Bus 连接。

| Manager 所属 | 定义位置 | 核心职责 |
|-------------|---------|---------|
| PID 1 | core/manager.h:143 | 管理所有 Unit 和 Job |
| logind | login/logind.h:25 | 管理 Session/Seat/User |
| networkd | network/networkd-manager.h:20 | 管理网络链路 |
| resolved | resolve/resolved-manager.h:35 | 管理 DNS 解析 |
| timesyncd | timesync/timesyncd-manager.h:37 | 管理 NTP 同步 |
| homed | home/homed-manager.h:28 | 管理用户家目录 |
| machined | machine/machined.h:19 | 管理虚拟机/容器 |
| portabled | portable/portabled.h:15 | 管理便携服务镜像 |
| oomd | oom/oomd-manager.h:32 | 管理 OOM 策略 |
| importd | import/importd.c:86 | 管理镜像导入 |
| udevd | udev/udevd.c | 管理设备事件 |
| wait-online | network/wait-online/manager.h:15 | 等待网络就绪 |

### 2.2 表驱动状态机

systemd 大量使用**表驱动状态机**来管理单元/服务的生命周期：

```c
// service.c 中的状态转换表示例
static const UnitActiveState state_translation_table[_SERVICE_STATE_MAX] = {
    [SERVICE_DEAD]             = UNIT_INACTIVE,
    [SERVICE_CONDITION]        = UNIT_ACTIVATING,
    [SERVICE_START_PRE]        = UNIT_ACTIVATING,
    [SERVICE_START]            = UNIT_ACTIVATING,
    [SERVICE_START_POST]       = UNIT_ACTIVATING,
    [SERVICE_RUNNING]          = UNIT_ACTIVE,
    [SERVICE_EXITED]           = UNIT_ACTIVE,
    [SERVICE_RELOAD]           = UNIT_RELOADING,
    [SERVICE_STOP]             = UNIT_DEACTIVATING,
    ...
};
```

这种模式也用于 Job 状态（`job.h:21-99`）、网络链路状态等。

### 2.3 单元类型虚函数表（vtable）

每种单元类型（service、socket、mount、timer 等）通过 `UnitVTable` 提供多态行为：

```c
// unit.h 中的虚函数表
const UnitVTable * const unit_vtable[_UNIT_TYPE_MAX] = {
    [UNIT_SERVICE]   = &service_vtable,
    [UNIT_SOCKET]    = &socket_vtable,
    [UNIT_TARGET]    = &target_vtable,
    [UNIT_DEVICE]    = &device_vtable,
    [UNIT_MOUNT]     = &mount_vtable,
    [UNIT_AUTOMOUNT] = &automount_vtable,
    [UNIT_SWAP]      = &swap_vtable,
    [UNIT_TIMER]     = &timer_vtable,
    [UNIT_PATH]      = &path_vtable,
    [UNIT_SLICE]     = &slice_vtable,
    [UNIT_SCOPE]     = &scope_vtable,
};
```

每个 vtable 包含 `load`、`start`、`stop`、`reload`、`serialize`、`deserialize` 等虚函数。

### 2.4 静态配置与运行时状态分离

systemd 严格分离静态配置（从配置文件加载）和运行时状态：

| 静态配置结构 | 运行时结构 | 用途 |
|------------|----------|------|
| ExecContext | ExecParameters / ExecRuntime | 进程执行环境 |
| Service（配置字段） | Service（PID/watchdog/fd_store） | 服务管理 |
| Network（.network文件） | Link（实际链路状态） | 网络管理 |

### 2.5 D-Bus 作为 IPC 核心

所有守护进程通过 **D-Bus** 对外暴露管理接口：
- PID 1 → `org.freedesktop.systemd1`
- logind → `org.freedesktop.login1`
- networkd → `org.freedesktop.network1`
- resolved → `org.freedesktop.resolve1`
- 等等

用户工具（systemctl、loginctl、networkctl）通过 D-Bus 与对应的守护进程通信。

### 2.6 Socket 激活

systemd 的标志性特性之一。守护进程可以延迟启动，直到有客户端连接到预先创建的 socket：

```
systemd-networkd.socket → 创建 rtnetlink socket
  ↓ 有连接时
systemd-networkd.service → 启动网络管理守护进程
```

## 三、代码组织结构

```
src/
├── core/          # PID 1 核心（Manager、Unit、Job、Execute）
├── systemd/       # 公共头文件（sd-bus.h、sd-event.h 等）
├── libsystemd/    # 公共库实现（sd-bus、sd-event、sd-journal 等）
├── libudev/       # libudev 兼容层
├── basic/         # 基础工具库（字符串、文件、哈希等）
├── shared/        # 共享代码（D-Bus辅助、安全、网络等）
├── fundamental/   # 最底层定义（可用于 EFI）
├── systemctl/     # systemctl 命令
├── journal/       # journald 日志守护进程
├── login/         # logind 登录管理
├── udev/          # udevd 设备管理
├── network/       # networkd 网络管理
├── resolve/       # resolved DNS 解析
├── timesync/      # timesyncd NTP 同步
├── home/          # homed 家目录管理
├── machine/       # machined 容器/VM 管理
├── portable/      # portabled 便携服务
├── nspawn/        # systemd-nspawn 容器工具
├── boot/          # systemd-boot EFI 启动器
├── oom/           # oomd OOM 处理
├── coredump/      # coredump 核心转储
├── sleep/         # 休眠/睡眠管理
├── cgroups-agent/ # cgroup 事件代理
└── ...            # 约 80 个其他子模块
```

## 四、构建系统

systemd 使用 **Meson** 构建系统：

```meson
project('systemd', 'c',
    version : '250',
    license : 'LGPLv2+',
    default_options: [
        'c_std=gnu11',
        'prefix=/usr',
        'sysconfdir=/etc',
        'localstatedir=/var',
    ],
    meson_version : '>= 0.53.2',
)
```

核心库依赖：
- **libseccomp** — 系统调用过滤（沙箱）
- **libpam** — PAM 认证
- **libaudit** — 审计日志
- **libkmod** — 内核模块管理
- **libmount/libblkid** — 挂载/块设备管理
- **libselinux/libapparmor** — 安全模块
- **libacl** — 访问控制列表

核心库 `libsystemd-core` 通过 `src/core/meson.build` 构建，链接了所有上述依赖，并通过 **gperf** 生成配置解析的查找表（`load-fragment-gperf.c`）。
