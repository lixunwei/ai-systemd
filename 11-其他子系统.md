# 其他子系统分析

## 一、systemd-homed — 家目录管理

### 1.1 概述

`systemd-homed` 管理用户家目录，支持加密、可移植的家目录。引入于 systemd 245。

D-Bus 名称：`org.freedesktop.home1`
源码位于：`src/home/`

### 1.2 核心数据结构

#### Home（homed-home.h:109-176）

```c
struct Home {
    Manager *manager;
    char *user_name;               // 用户名
    uid_t uid;                     // UID

    HomeState state;               // 家目录状态
    int signed_locally;            // 是否本地签名

    UserRecord *record;            // 用户记录
    Secret *secret;                // 认证密钥

    // 工作进程
    pid_t worker_pid;              // 后台工作进程
    int worker_stdout_fd;          // 工作进程输出

    // 挂载/加密
    // 通过 HomeSetup 管理

    // 操作队列
    LIST_HEAD(Operation, pending_operations);

    // 会话管理
    unsigned n_pin_fds;            // pin 引用计数
    bool locked;                   // 是否已锁定
    ...
};
```

### 1.3 存储后端

| 后端 | 说明 |
|------|------|
| LUKS | LUKS2 加密卷（推荐，最完整） |
| fscrypt | 文件系统级加密（ext4/f2fs） |
| directory | 普通目录 |
| subvolume | Btrfs 子卷 |
| CIFS | 网络共享（SMB） |

### 1.4 生命周期

```
homectl create user1 --storage=luks
  ↓
homed 创建 LUKS 卷 + 用户记录
  ↓
用户登录 → PAM 调用 homed
  ↓
home_activate()
  ├── 打开 LUKS 设备
  ├── 挂载文件系统到 /home/user1
  ├── 验证密码/FIDO2/PKCS#11
  └── 更新用户数据库
  ↓
用户退出
  ↓
home_deactivate()
  ├── 卸载文件系统
  └── 关闭 LUKS 设备
```

### 1.5 管理命令（homectl）

```bash
homectl create user1                 # 创建用户
homectl remove user1                 # 删除用户
homectl activate user1               # 激活家目录
homectl deactivate user1             # 停用家目录
homectl lock user1                   # 锁定（屏幕锁定时）
homectl unlock user1                 # 解锁
homectl resize user1 50G             # 调整大小
homectl passwd user1                 # 修改密码
homectl inspect user1                # 查看详情
homectl with user1 -- command        # 在用户上下文中运行命令
```

## 二、systemd-oomd — OOM 守护进程

### 2.1 概述

`systemd-oomd` 基于 PSI（Pressure Stall Information）和 cgroup v2 的内存管理，在内存压力过大时主动终止进程组，防止系统完全卡死。

D-Bus 名称：`org.freedesktop.oom1`
源码位于：`src/oom/`

### 2.2 Manager 结构体（oomd-manager.h:32-62）

```c
struct Manager {
    sd_bus *bus;
    sd_event *event;

    // 配置
    bool dry_run;                     // 干运行模式（仅日志不杀）
    int swap_used_limit_permyriad;    // 交换使用限制（万分比）
    int mem_pressure_limit_permyriad; // 内存压力限制
    usec_t mem_pressure_duration_usec; // 压力持续时间阈值

    // 监控目标
    Hashmap *polkit_registry;
    OomdCGroupContext *system_context;   // 系统级上下文

    // 被监控的 cgroup
    Hashmap *monitored_swap_cgroup_contexts;     // 监控交换的 cgroup
    Hashmap *monitored_mem_pressure_cgroup_contexts; // 监控内存压力的 cgroup

    // Varlink
    VarlinkServer *varlink_server;
    ...
};
```

### 2.3 工作原理

```
定期检查（默认每 1 秒）
  ↓
检查系统级 PSI 内存压力
  ├── some 压力 > 阈值 且 持续 > duration？
  │     → 选择内存使用最高的 cgroup
  │     → SIGKILL 该 cgroup 下的所有进程
  │
  ├── swap 使用 > 限制？
  │     → 选择 swap 使用最高的 cgroup
  │     → SIGKILL 该 cgroup 下的所有进程
  │
  └── 检查各个被监控 cgroup 的 PSI
        → 如果某个 cgroup 内存压力过高
        → 选择其子 cgroup 中内存最高的终止
```

### 2.4 配置

```ini
# /etc/systemd/oomd.conf
[OOM]
SwapUsedLimit=90%
DefaultMemoryPressureLimit=60%
DefaultMemoryPressureDurationSec=30s
```

单元文件中启用监控：
```ini
[Service]
ManagedOOMSwap=kill          # 交换压力时可被杀
ManagedOOMMemoryPressure=kill # 内存压力时可被杀
ManagedOOMMemoryPressureLimit=80%
```

## 三、coredump — 核心转储处理

### 3.1 概述

`systemd-coredump` 接收内核产生的核心转储，存储并索引它们，方便后续分析。

源码位于：`src/coredump/`

### 3.2 核心文件

| 文件 | 职责 |
|------|------|
| coredump.c | coredump 处理守护进程 |
| coredumpctl.c | coredumpctl 命令行工具 |
| coredump-vacuum.c | 空间回收 |

### 3.3 处理流程

```
进程崩溃
  ↓
内核检查 /proc/sys/kernel/core_pattern
  = |/usr/lib/systemd/systemd-coredump ...
  ↓
内核通过管道传递 coredump 数据
  ↓
systemd-coredump@.service 实例化启动
  ↓
coredump.c → process_kernel()
  ├── 接收元数据（PID、UID、信号、可执行文件等）
  ├── 接收 coredump 数据
  ├── 压缩（LZ4/XZ）
  ├── 存储
  │     ├── external: /var/lib/systemd/coredump/
  │     └── journal: 嵌入到 journal 日志中
  ├── 记录日志条目
  │     MESSAGE_ID=fc2e22bc6ee647b6b90729ab34a250b1
  │     COREDUMP_PID=, COREDUMP_UID=, COREDUMP_GID=
  │     COREDUMP_SIGNAL=, COREDUMP_EXE=, COREDUMP_COMM=
  └── 空间回收（如果需要）
```

### 3.4 配置

```ini
# /etc/systemd/coredump.conf
[Coredump]
Storage=external            # none/external/journal
Compress=yes
ProcessSizeMax=2G           # 最大处理大小
ExternalSizeMax=2G          # 外部存储最大大小
JournalSizeMax=767M         # Journal 内最大大小
MaxUse=                     # 总空间限制
KeepFree=                   # 保留空闲空间
```

### 3.5 coredumpctl

```bash
coredumpctl list              # 列出所有 coredump
coredumpctl info PID          # 查看 coredump 详情
coredumpctl dump PID          # 导出 coredump 文件
coredumpctl gdb PID           # 直接用 GDB 打开
coredumpctl debug PID         # 同 gdb
```

## 五、systemd-sleep — 睡眠/休眠

### 5.1 概述

处理系统挂起（suspend）、冬眠（hibernate）等电源状态转换。

源码位于：`src/sleep/`

### 5.2 处理流程

```
logind 触发睡眠请求
  ↓
启动 systemd-suspend.service / systemd-hibernate.service
  ↓
sleep.c → execute()
  ├── 锁定所有 homed 家目录
  │     D-Bus → homed.LockAllHomes()
  ├── 写入冬眠位置信息（如果冬眠）
  │     /sys/power/resume
  │     /sys/power/resume_offset
  ├── 写入模式
  │     /sys/power/disk = "platform"/"shutdown"/...
  ├── 写入状态触发
  │     echo "mem" > /sys/power/state （挂起）
  │     echo "disk" > /sys/power/state （冬眠）
  ├── === 系统睡眠 ===
  ├── 唤醒后继续执行
  └── 返回成功
```

### 5.3 睡眠类型

| 类型 | 单元 | 效果 |
|------|------|------|
| suspend | systemd-suspend.service | 挂起到内存 |
| hibernate | systemd-hibernate.service | 冬眠到磁盘 |
| hybrid-sleep | systemd-hybrid-sleep.service | 混合（内存+磁盘） |
| suspend-then-hibernate | systemd-suspend-then-hibernate.service | 先挂起，超时后冬眠 |

### 5.4 配置

```ini
# /etc/systemd/sleep.conf
[Sleep]
AllowSuspend=yes
AllowHibernation=yes
AllowSuspendThenHibernate=yes
AllowHybridSleep=yes
SuspendMode=
SuspendState=mem standby freeze
HibernateMode=platform shutdown
HibernateState=disk
HibernateDelaySec=120min
```

## 六、cgroups-agent — cgroup 事件代理

### 6.1 概述

`cgroups-agent` 是一个极简的辅助程序（`cgroups-agent.c`，约 40 行），用于向 PID 1 发送 cgroup 空事件通知。

```c
// 核心逻辑（简化）
int main(int argc, char *argv[]) {
    // 创建 datagram socket
    // 连接到 /run/systemd/cgroups-agent
    // 发送 argv[1]（cgroup 路径）
    return 0;
}
```

当 cgroup 中最后一个进程退出时，内核通过 release_agent 调用此程序，通知 PID 1 该 cgroup 已空（主要用于 cgroup v1）。cgroup v2 使用 inotify 替代此机制。
