## 一、Job 合并矩阵——精确的代码实现

### 1.1 合并查找表

合并矩阵定义在 `job.c:359-367`，使用**下三角矩阵**存储（利用合并操作的交换律）：

```c
static const JobType job_merging_table[] = {
/* What \ With       *  START        VERIFY_ACTIVE   STOP    RELOAD */
/*                   */
/*START              */
/*VERIFY_ACTIVE      */ JOB_START,
/*STOP               */ -1,           -1,
/*RELOAD             */ JOB_RELOAD_OR_START,  JOB_RELOAD,  -1,
/*RESTART            */ JOB_RESTART,  JOB_RESTART,    -1,    JOB_RESTART,
};
```

完整合并结果矩阵（对称）：

```
           │ START  │ VERIFY │  STOP  │ RELOAD │RESTART │
    ───────┼────────┼────────┼────────┼────────┼────────┤
    START  │ START  │ START  │   ✗    │RELOAD+ │RESTART │
    VERIFY │ START  │ VERIFY │   ✗    │ RELOAD │RESTART │
    STOP   │   ✗    │   ✗    │  STOP  │   ✗    │   ✗    │
    RELOAD │RELOAD+ │ RELOAD │   ✗    │ RELOAD │RESTART │
    RESTART│RESTART │RESTART │   ✗    │RESTART │RESTART │

    ✗ = 冲突，无法合并（返回 -1）
    RELOAD+ = JOB_RELOAD_OR_START
```

### 1.2 索引计算

查找下三角矩阵使用公式（`job.c:383`）：

```c
// a > b（保证 a 是较大的值）
index = (a - 1) * a / 2 + b
```

例如 RESTART(4) 与 RELOAD(3) 的索引：`(4-1)*4/2 + 3 = 9` → 取 `job_merging_table[9]` = `JOB_RESTART`。

### 1.3 合并后的折叠（Collapse）

合并可能产生中间类型 `JOB_RELOAD_OR_START`、`JOB_TRY_RESTART`、`JOB_TRY_RELOAD`，需要根据**单元当前状态**折叠为实际可执行类型（`job.c:414-443`）：

```c
JobType job_type_collapse(JobType t, Unit *u) {
    UnitActiveState s;
    switch (t) {
        case JOB_TRY_RESTART:
            s = unit_active_state(u);
            if (!UNIT_IS_ACTIVE_OR_RELOADING(s))
                return JOB_NOP;    // 没在运行 → 什么都不做
            return JOB_RESTART;    // 在运行 → 重启

        case JOB_TRY_RELOAD:
            s = unit_active_state(u);
            if (!UNIT_IS_ACTIVE_OR_RELOADING(s))
                return JOB_NOP;    // 没在运行 → 什么都不做
            return JOB_RELOAD;     // 在运行 → 重载

        case JOB_RELOAD_OR_START:
            s = unit_active_state(u);
            if (!UNIT_IS_ACTIVE_OR_RELOADING(s))
                return JOB_START;  // 没在运行 → 启动
            return JOB_RELOAD;     // 在运行 → 重载

        default:
            return t;              // 其他类型不变
    }
}
```

**关键设计**：`merge_and_collapse` 是一个原子操作（`job.c:445-454`），先合并再折叠，确保中间类型不泄漏到外部。

### 1.4 冗余判断

Job 是否冗余（已经满足）由 `job_type_is_redundant()` 判断（`job.c:386-412`）：

```c
bool job_type_is_redundant(JobType a, UnitActiveState b) {
    switch (a) {
        case JOB_START:
            return IN_SET(b, UNIT_ACTIVE, UNIT_RELOADING);
            // 已 active → start 是多余的

        case JOB_STOP:
            return IN_SET(b, UNIT_INACTIVE, UNIT_FAILED);
            // 已 inactive/failed → stop 是多余的

        case JOB_VERIFY_ACTIVE:
            return IN_SET(b, UNIT_ACTIVE, UNIT_RELOADING);
            // 已 active → verify 满足

        case JOB_RELOAD:
            return b == UNIT_RELOADING;
            // 正在 reloading → reload 多余

        case JOB_RESTART:
            return b == UNIT_ACTIVATING;
            // 正在 activating → restart 多余

        case JOB_NOP:
            return true;  // NOP 永远是多余的
    }
}
```

## 二、循环依赖检测算法——逐行解析

### 2.1 算法概述

循环检测使用**基于 DFS 的着色标记算法**，实现在 `transaction_verify_order_one()`（`transaction.c:349-479`）。

### 2.2 算法数据结构

每个 Job 有两个辅助字段：
```c
struct Job {
    unsigned generation;  // 当前遍历的"代"标记
    Job *marker;          // 回溯指针（形成路径链）
    ...
};
```

### 2.3 算法步骤图解

```
初始状态：所有 Job 的 generation=0, marker=NULL

开始遍历 Job A：
  A.generation = G
  A.marker = A (自己指向自己=路径起点)
  │
  ├─ 检查 A 的 BEFORE 方向邻居
  │   找到 B (B.After=A 意味着 A 必须在 B 之前)
  │   B 有 Job 且 job_compare 确认 A 在 B 之前
  │   └─ 递归进入 B
  │       B.generation = G
  │       B.marker = A (回溯指向 A)
  │       │
  │       ├─ 检查 B 的 BEFORE 方向邻居
  │       │   找到 C (C.After=B)
  │       │   └─ 递归进入 C
  │       │       C.generation = G
  │       │       C.marker = B
  │       │       │
  │       │       ├─ 检查 C 的邻居
  │       │       │   找到 A ← A.generation == G ← ⚠️ 发现循环!
  │       │       │   │
  │       │       │   └─ 循环处理（见下文）
  │       │       │
  │       │       └─ 回溯：C.marker = NULL
  │       │
  │       └─ 回溯：B.marker = NULL
  │
  └─ 回溯：A.marker = NULL
```

### 2.4 循环打破策略

当检测到循环（`j.generation == generation`）时（`transaction.c:366-435`）：

```
发现循环，当前 j=A，from=C
  │
  ├── 步骤1：沿 marker 回溯，收集循环路径
  │   k=C → k.marker=B → k.marker=A → k==j，循环完整
  │   路径: [C/start, B/start, A/start]
  │
  ├── 步骤2：在路径中寻找可删除的 Job
  │   条件: 在事务中 AND !matters_to_anchor
  │   优先删除不重要的 Job
  │
  ├── 步骤3a（找到可删除的 Job）：
  │   transaction_delete_unit(tr, delete->unit)
  │   → 删除该 Unit 的所有 Job 及其依赖
  │   → 返回 -EAGAIN
  │   → 外层循环重新运行整个验证
  │
  └── 步骤3b（所有 Job 都重要）：
      → 循环不可打破
      → 返回错误 "Transaction order is cyclic"
```

### 2.5 排序方向的判断

循环检测需要知道**两个 Job 谁先执行**。这由 `job_compare()`（`job.c:1615-1638`）决定：

```c
int job_compare(Job *a, Job *b, UnitDependencyAtom assume_dep) {
    // NOP 和 ignore_order 不参与排序
    if (a->type == JOB_NOP || b->type == JOB_NOP) return 0;
    if (a->ignore_order || b->ignore_order) return 0;

    // After 方向的比较转换为 Before 方向（对称性）
    if (assume_dep == UNIT_ATOM_AFTER)
        return -job_compare(b, a, UNIT_ATOM_BEFORE);

    // assume_dep == UNIT_ATOM_BEFORE
    // 即 "a 的 Before 列表中有 b" = "a Before b" = "a 应先于 b"

    // 核心规则：STOP/RESTART 总是先执行
    if (IN_SET(b->type, JOB_STOP, JOB_RESTART))
        return 1;   // b 是 stop → b 先执行 → a 在 b 之后
    else
        return -1;  // b 是 start/reload → a 先执行
}
```

**返回值含义**：
- `< 0`：a 应在 b 之前执行
- `> 0`：a 应在 b 之后执行
- `= 0`：无排序关系

**排序真值表**（假设 a.Before=b）：

| a 类型 | b 类型 | 结果 | 含义 |
|--------|--------|------|------|
| start | start | a < b | a 先启动 |
| start | stop | a > b | b 先停止，然后 a 启动 |
| start | restart | a > b | b 先 restart |
| stop | start | a < b | a 先停止 |
| stop | stop | a > b | b 先停止（依赖的先停） |
| stop | restart | a > b | b 先 restart |
| reload | start | a < b | a 先 reload |
| reload | stop | a > b | b 先停止 |
| restart | start | a < b | a 先 restart |
| restart | stop | a > b | b 先停止 |

## 三、可运行性判断——精确的调度条件

`job_is_runnable()`（`job.c:456-496`）决定一个 Job 是否可以立即执行：

```c
static bool job_is_runnable(Job *j) {
    Unit *other;

    // 忽略排序的 Job 总是可运行
    if (j->ignore_order) return true;

    // NOP 总是可运行
    if (j->type == JOB_NOP) return true;

    // 检查 AFTER 方向：j 需要在这些单元之后执行
    UNIT_FOREACH_DEPENDENCY(other, j->unit, UNIT_ATOM_AFTER)
        if (other->job && job_compare(j, other->job, UNIT_ATOM_AFTER) > 0) {
            log_unit_debug(j->unit,
                "starting held back, waiting for: %s", other->id);
            return false;  // 还有 Job 需要先执行
        }

    // 检查 BEFORE 方向：j 需要在这些单元之前执行
    UNIT_FOREACH_DEPENDENCY(other, j->unit, UNIT_ATOM_BEFORE)
        if (other->job && job_compare(j, other->job, UNIT_ATOM_BEFORE) > 0) {
            log_unit_debug(j->unit,
                "stopping held back, waiting for: %s", other->id);
            return false;  // 还有 Job 需要先执行
        }

    return true;  // 所有排序依赖已满足
}
```

**核心逻辑**：遍历当前 Job 的所有排序邻居（AFTER 和 BEFORE 方向），如果任何邻居有活跃 Job 且该 Job 应当先于当前 Job 执行（`job_compare() > 0`），则当前 Job 不可运行。

## 四、RESTART 的两阶段实现

RESTART 不是一个原子操作，而是**先 STOP 再 START** 的两阶段过程（`job.c:799-827, 930-961`）：

### 阶段 1：STOP

```c
// job_perform_on_unit() 中
case JOB_RESTART:
    t = JOB_STOP;      // 把 restart 伪装成 stop
    _fallthrough_;      // 然后执行 stop 逻辑
case JOB_STOP:
    r = unit_stop(u);   // 执行停止
    break;
```

### 阶段 2：状态转换（STOP 完成后）

```c
// job_finish_and_invalidate() 中
if (result == JOB_DONE && t == JOB_RESTART) {
    // stop 阶段成功完成
    job_change_type(j, JOB_START);   // 转换为 START
    job_set_state(j, JOB_WAITING);   // 回到 WAITING 状态
    job_add_to_run_queue(j);         // 重新加入运行队列
    goto finish;                      // 不释放 Job，等待 start
}
```

**流程图**：

```
RESTART Job 安装
  │
  ▼ [phase 1]
Job 类型 = RESTART, 但执行 unit_stop()
  │
  ▼ SIGCHLD / 状态变更
job_finish_and_invalidate(j, JOB_DONE)
  │ 检测到 type==RESTART && result==DONE
  │
  ▼ [phase 2]
job_change_type(j, JOB_START)
job_set_state(j, JOB_WAITING)
job_add_to_run_queue(j)
  │
  ▼ run_queue 调度
job_run_and_invalidate(j)  → type==START → unit_start()
  │
  ▼ 启动成功
job_finish_and_invalidate(j, JOB_DONE)
  → Job 完成，释放
```

## 五、失败传播机制

### 5.1 传播规则

当 Job 失败时，`job_finish_and_invalidate()` 负责传播（`job.c:969-1011`）：

```c
// 失败且需要递归传播
if (result != JOB_DONE && recursive) {
    if (IN_SET(t, JOB_START, JOB_VERIFY_ACTIVE))
        // START/VERIFY 失败 → 传播给 PROPAGATE_START_FAILURE 依赖
        job_fail_dependencies(u, UNIT_ATOM_PROPAGATE_START_FAILURE);
    else if (t == JOB_STOP)
        // STOP 失败 → 传播给 PROPAGATE_STOP_FAILURE 依赖
        job_fail_dependencies(u, UNIT_ATOM_PROPAGATE_STOP_FAILURE);
}
```

### 5.2 传播方向

```
A (Requires=B) 的 START 失败
  │
  ▼
遍历 B 的 PROPAGATE_START_FAILURE 原子的依赖
  → 即所有 Requires=B / BindsTo=B 的单元
  → 这些单元的 START/VERIFY_ACTIVE Job 也被标记为 JOB_DEPENDENCY
```

### 5.3 特殊的"不活跃即失败"传播

对于 oneshot 类型服务（启动后立即退出进入 inactive），还有额外检查：

```c
// 即使 result==JOB_DONE，如果单元不活跃也需要传播
if (result == JOB_DONE && recursive &&
    IN_SET(t, JOB_START, JOB_RELOAD) &&
    !UNIT_IS_ACTIVE_OR_RELOADING(unit_active_state(u)))
    job_fail_dependencies(u, UNIT_ATOM_PROPAGATE_INACTIVE_START_AS_FAILURE);
```

## 六、事务中不可合并冲突的解决

### 6.1 冲突场景

典型场景：事务中同一 Unit 同时出现 START 和 STOP（合并表中为 -1）。

### 6.2 解决策略（`delete_one_unmergeable_job()`，`transaction.c:141-217`）

```
找到两个不可合并的 Job j 和 k
  │
  ├── 两个都不重要（!matters_to_anchor）
  │     ├── j 是 STOP 且由 Conflicts 拉入
  │     │     → 删除 k（保留冲突引起的 stop）
  │     ├── j 是 STOP 但非冲突引起
  │     │     → 删除 j（倾向保留 start）
  │     ├── k 是 STOP 且由 Conflicts 拉入
  │     │     → 删除 j
  │     ├── k 是 STOP 但非冲突引起
  │     │     → 删除 k
  │     └── 都不是 STOP
  │           → 删除 j（默认选择）
  │
  ├── 仅 j 不重要
  │     → 删除 j
  │
  ├── 仅 k 不重要
  │     → 删除 k
  │
  └── 都重要
       → 返回 -ENOEXEC（无法解决）
       → 上层报错 "Transaction contains conflicting jobs"
```

**设计哲学**：
- **倾向保留 START**（正向操作），删除 STOP
- **例外**：如果 STOP 是因为 Conflicts 关系拉入的，则保留 STOP
- **锚点保护**：matters_to_anchor 的 Job 绝不删除

## 七、事务中的 GC（垃圾回收）

`transaction_collect_garbage()`（`transaction.c:503-533`）删除孤立 Job：

```c
// 循环扫描直到没有变化
do {
    again = false;
    HASHMAP_FOREACH(j, tr->jobs) {
        // 锚点 Job 不回收
        if (tr->anchor_job == j)
            continue;

        // 如果没有其他 Job 依赖此 Job → 垃圾
        if (!j->object_list) {
            transaction_delete_job(tr, j, true);
            again = true;  // 删除后可能产生新的孤立 Job
            break;         // 重新扫描
        }
    }
} while (again);
```

**原理**：`JobDependency` 链表 `object_list` 记录了哪些 Job 依赖此 Job。如果没有任何 Job 依赖它（且它不是锚点），则它是垃圾。

## 八、Job 完成后的连锁调度

当一个 Job 完成时，需要唤醒等待它的其他 Job（`job.c:1016-1026`）：

```c
finish:
    // 唤醒所有 AFTER 方向的 Job
    UNIT_FOREACH_DEPENDENCY(other, u, UNIT_ATOM_AFTER)
        if (other->job) {
            job_add_to_run_queue(other->job);  // 重新评估可运行性
            job_add_to_gc_queue(other->job);
        }

    // 唤醒所有 BEFORE 方向的 Job
    UNIT_FOREACH_DEPENDENCY(other, u, UNIT_ATOM_BEFORE)
        if (other->job) {
            job_add_to_run_queue(other->job);
            job_add_to_gc_queue(other->job);
        }

    manager_check_finished(u->manager);  // 检查是否所有 Job 都完成了
```

**执行链示例**：

```
1. START network.target 完成 (JOB_DONE)
   │
   └─ 唤醒所有 After=network.target 的 Job:
      ├── START sshd.service  → 加入 run_queue → 可运行 → 执行
      ├── START nginx.service → 加入 run_queue → 可运行 → 执行
      └── START postfix.service → 加入 run_queue → 检查还有其他依赖...

2. START sshd.service 完成 (JOB_DONE)
   │
   └─ 唤醒所有依赖 sshd 的 Job（如果有的话）
      → manager_check_finished() → 如果所有 Job 完成 → 启动完成
```

## 九、Job 超时机制

每个 Job 都有超时定时器（`job.c:1033-1070`）：

```c
static int job_dispatch_timer(sd_event_source *s, uint64_t monotonic, void *userdata) {
    Job *j = userdata;

    log_unit_warning(j->unit,
        "Job %s/%s timed out.", j->unit->id, job_type_to_string(j->type));

    // 超时处理
    u = j->unit;
    job_finish_and_invalidate(j, JOB_TIMEOUT, true, false);
    // result=JOB_TIMEOUT 会触发失败传播

    // 如果是 START 超时，emergency_action() 可能触发紧急操作
    // （如 StartLimitAction=reboot）
    emergency_action(u->manager, ...);
}
```

超时时间配置：
```ini
[Service]
JobTimeoutSec=5min        # Job 总超时
JobRunningTimeoutSec=3min # Job 运行阶段超时
TimeoutStartSec=90s       # 启动超时
TimeoutStopSec=90s        # 停止超时
```

## 十、完整数据流图

```
用户请求: systemctl start foo.service
  │
  ▼
manager_add_job(START, foo)
  │
  ├── Transaction *tr = transaction_new()
  │
  ├── transaction_add_job_and_dependencies(tr, START, foo)
  │     │
  │     ├── 创建 Job(START, foo) = anchor_job
  │     │
  │     ├── 递归展开 Requires/Wants/Conflicts 依赖
  │     │     PULL_IN_START     → 添加 START Job（必须成功）
  │     │     PULL_IN_START_IGN → 添加 START Job（可失败）
  │     │     PULL_IN_VERIFY    → 添加 VERIFY Job
  │     │     PULL_IN_STOP      → 添加 STOP Job（冲突）
  │     │
  │     └── 每个递归 Job 都通过 JobDependency 链接到父 Job
  │
  ├── transaction_activate(tr)
  │     │
  │     ├── [1] find_jobs_that_matter_to_anchor  ← DFS 标记重要性
  │     ├── [2] minimize_impact                  ← 保护运行中服务
  │     ├── [3] drop_redundant                   ← 删除已满足的 Job
  │     ├── [4-5] collect_garbage + verify_order  ← GC + 循环检测（循环）
  │     ├── [6-7] merge_jobs + collect_garbage    ← 合并 + GC（循环）
  │     ├── [8] drop_redundant                   ← 再次去冗余
  │     ├── [9] is_destructive                   ← 破坏性检查
  │     └── [10] apply                           ← 安装到 Manager
  │           │
  │           ├── job_install(j)        → j 挂到 unit->job
  │           ├── job_add_to_run_queue  → j 进入 Manager.run_queue
  │           └── job_start_timer       → 启动超时计时器
  │
  ├── transaction_free(tr)
  │
  ▼
Manager 事件循环
  │
  ├── manager_dispatch_run_queue()
  │     │
  │     ├── j = prioq_peek(run_queue)
  │     ├── job_is_runnable(j)?
  │     │     检查所有排序邻居的 Job 是否完成
  │     │     → 是 → 继续
  │     │     → 否 → 跳过，等待
  │     │
  │     └── job_run_and_invalidate(j)
  │           │
  │           ├── job_set_state(JOB_RUNNING)
  │           │
  │           ├── START  → unit_start(u)  → fork()+exec()
  │           ├── STOP   → unit_stop(u)   → kill()
  │           ├── RELOAD → unit_reload(u) → SIGHUP
  │           ├── RESTART→ unit_stop(u)   → [phase 1]
  │           │
  │           └── 处理返回值
  │                 -EAGAIN  → 回到 WAITING（还没准备好）
  │                 -EALREADY→ JOB_DONE（已经是目标状态）
  │                 成功     → 等待异步完成
  │
  ├── SIGCHLD / sd_notify 事件
  │     │
  │     └── unit 状态变更 → service_sigchld_event()
  │           → job_finish_and_invalidate(j, result)
  │                │
  │                ├── JOB_DONE
  │                │     ├── 如果是 RESTART → 转换为 START，重入 run_queue
  │                │     └── 否则 → 释放 Job，唤醒等待的 Job
  │                │
  │                ├── JOB_FAILED / JOB_TIMEOUT / JOB_DEPENDENCY
  │                │     ├── 传播失败给依赖 Job
  │                │     ├── 触发 OnFailure= 单元
  │                │     └── 释放 Job
  │                │
  │                └── 唤醒排序依赖的 Job
  │                      → job_add_to_run_queue(依赖的 job)
  │
  └── 所有 Job 完成 → manager_check_finished()
```
