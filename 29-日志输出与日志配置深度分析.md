# 29. 日志输出与日志配置深度分析

## 1. 概述

systemd 为服务进程提供了灵活的**日志输出路由**机制，可以将 stdout/stderr 定向到多种目标（journal、文件、tty、kmsg 等），并通过结构化字段、速率限制、级别过滤等实现精细控制。

### 核心源文件

| 文件 | 关键行 | 职责 |
|------|--------|------|
| `src/core/execute.c` | 274-747 | 输出目标设置、journal 流连接 |
| `src/core/execute.h` | ExecContext 字段 | 日志相关配置存储 |
| `src/core/load-fragment.c` | 1272-1385 | StandardOutput/Error 解析 |
| `src/journal/journald-stream.c` | 73-445 | journal 流协议状态机 |
| `src/journal/journald-server.c` | 927-1121 | 速率限制、额外字段注入 |
| `src/journal/journald-context.c` | 120-133 | ClientContext 默认值 |

## 2. ExecOutput 类型

### 2.1 枚举定义

```c
typedef enum ExecOutput {
    EXEC_OUTPUT_INHERIT,          // 继承父进程（stdin 的复制）
    EXEC_OUTPUT_NULL,             // /dev/null
    EXEC_OUTPUT_TTY,              // 指定 TTY 设备
    EXEC_OUTPUT_JOURNAL,          // journal 守护进程（结构化）
    EXEC_OUTPUT_KMSG,             // 内核环形缓冲区
    EXEC_OUTPUT_JOURNAL_AND_CONSOLE, // journal + /dev/console
    EXEC_OUTPUT_KMSG_AND_CONSOLE,    // kmsg + /dev/console
    EXEC_OUTPUT_SOCKET,           // 激活 socket（用于 inetd 风格）
    EXEC_OUTPUT_NAMED_FD,         // 命名 fd（FileDescriptorStoreMax）
    EXEC_OUTPUT_FILE,             // 写入文件（覆盖）
    EXEC_OUTPUT_FILE_APPEND,      // 追加写入文件
    EXEC_OUTPUT_FILE_TRUNCATE,    // 截断后写入文件
    _EXEC_OUTPUT_MAX,
} ExecOutput;
```

### 2.2 配置映射

| 配置值 | ExecOutput | 行为 |
|--------|-----------|------|
| `inherit` | INHERIT | 继承 stdin 的 fd |
| `null` | NULL | 丢弃所有输出 |
| `tty` | TTY | 输出到 TTYPath= 指定的终端 |
| `journal` | JOURNAL | 通过 socket 发送到 journald |
| `kmsg` | KMSG | 写入 /dev/kmsg |
| `journal+console` | JOURNAL_AND_CONSOLE | 同时输出到 journal 和控制台 |
| `kmsg+console` | KMSG_AND_CONSOLE | 同时输出到 kmsg 和控制台 |
| `socket` | SOCKET | 复用激活 socket |
| `fd:name` | NAMED_FD | 复用命名 fd |
| `file:/path` | FILE | 覆盖写入指定文件 |
| `append:/path` | FILE_APPEND | 追加写入指定文件 |
| `truncate:/path` | FILE_TRUNCATE | 截断后写入 |

## 3. setup_output() 实现（execute.c:587-747）

```c
static int setup_output(Unit *unit, ExecContext *context,
                        ExecParameters *params,
                        int fileno, int socket_fd,
                        const char *ident, uid_t uid, gid_t gid):
  switch (output):
    case EXEC_OUTPUT_INHERIT:
      // 特殊处理：stdout 继承 stdin，stderr 继承 stdout
      if (fileno == STDOUT_FILENO && stdin 是 null/socket)
        → 回退到 journal
      else
        return dup2(STDIN_FILENO, fileno);

    case EXEC_OUTPUT_NULL:
      return open_null_as(O_WRONLY, fileno);

    case EXEC_OUTPUT_TTY:
      return open_terminal_as(tty_path(), O_WRONLY, fileno);

    case EXEC_OUTPUT_JOURNAL:
    case EXEC_OUTPUT_KMSG:
    case EXEC_OUTPUT_JOURNAL_AND_CONSOLE:
    case EXEC_OUTPUT_KMSG_AND_CONSOLE:
      return connect_logger_as(unit, context, params, output,
                               ident, fileno, uid, gid);

    case EXEC_OUTPUT_SOCKET:
      return dup2(socket_fd, fileno);

    case EXEC_OUTPUT_NAMED_FD:
      fd = exec_params_get_named_fd(params, context->stdio_fdname[fileno]);
      return move_fd(fd, fileno, 0);

    case EXEC_OUTPUT_FILE:
    case EXEC_OUTPUT_FILE_APPEND:
    case EXEC_OUTPUT_FILE_TRUNCATE:
      flags = O_WRONLY | (output == APPEND ? O_APPEND :
                          output == TRUNCATE ? O_TRUNC : 0);
      fd = acquire_path(context->stdio_file[fileno], flags, 0644);
      return move_fd(fd, fileno, 0);
```

## 4. Journal 流协议（execute.c:274-375 + journald-stream.c）

### 4.1 连接流程

```c
connect_logger_as(unit, context, params, output, ident, fileno, uid, gid):
  1. 连接 UNIX socket:
     路径: /run/systemd/journal/stdout
     (或命名空间: /run/systemd/journal.<namespace>/stdout)

  2. 发送设置头（换行分隔的纯文本）:
     line 1: unit_id (如 "nginx.service")
     line 2: syslog_identifier (如 "nginx")
     line 3: syslog_priority (如 "30" = LOG_DAEMON|LOG_INFO)
     line 4: level_prefix (如 "1" = 解析 "<N>" 前缀)
     line 5: forward_to_syslog (如 "0")
     line 6: forward_to_kmsg (如 "0")
     line 7: forward_to_console (如 "1" if +console)

  3. 将 socket fd dup2 到 stdout/stderr
```

### 4.2 journald 端状态机（journald-stream.c:73-105）

```c
typedef enum StdoutStreamState {
    STDOUT_STREAM_IDENTIFIER,      // 等待 unit_id
    STDOUT_STREAM_UNIT_ID,         // 等待 syslog_identifier
    STDOUT_STREAM_PRIORITY,        // 等待 priority
    STDOUT_STREAM_LEVEL_PREFIX,    // 等待 level_prefix 标志
    STDOUT_STREAM_FORWARD_TO_SYSLOG,
    STDOUT_STREAM_FORWARD_TO_KMSG,
    STDOUT_STREAM_FORWARD_TO_CONSOLE,
    STDOUT_STREAM_RUNNING,         // 正常接收日志行
} StdoutStreamState;
```

每收到一行文本，journald 按状态逐步解析设置头，进入 RUNNING 后每一行成为一条日志条目。

### 4.3 Level Prefix 解析

当 `level_prefix=1` 时，服务输出可以用 `<N>` 前缀指定日志级别：

```
<3>Error: something failed     → LOG_ERR
<6>Info: operation completed   → LOG_INFO
<7>Debug: internal state       → LOG_DEBUG
Normal line without prefix     → 使用默认 priority
```

### 4.4 JOURNAL_STREAM 环境变量

```c
// exec_child 中设置:
JOURNAL_STREAM=<device>:<inode>
// 值为 stdout fd 的 stat 结果
// 用途: 库代码可检测 stdout 是否已连接到 journal
//       sd_journal_stream_fd() 等 API 使用此信息
```

## 5. LogLevelMax=

### 5.1 配置

```ini
[Service]
LogLevelMax=notice    # 只记录 notice 及以上级别
```

### 5.2 实现

```c
// 存储在 ExecContext.log_level_max（默认 -1 = 不限制）

// PID 1 侧: unit_log_level_test()
if (log_level_max >= 0 && LOG_PRI(level) > log_level_max)
  return false;  // 丢弃该日志

// journald 侧: client_context_test_priority()
if (c->log_level_max >= 0 && LOG_PRI(priority) > c->log_level_max)
  return false;  // 丢弃
```

级别映射：
| 值 | 级别 | 含义 |
|----|------|------|
| 0 | emerg | 系统不可用 |
| 1 | alert | 必须立即处理 |
| 2 | crit | 严重条件 |
| 3 | err | 错误 |
| 4 | warning | 警告 |
| 5 | notice | 正常但重要 |
| 6 | info | 信息 |
| 7 | debug | 调试 |

## 6. LogRateLimitIntervalSec= / LogRateLimitBurst=

### 6.1 配置

```ini
[Service]
LogRateLimitIntervalSec=30s    # 30秒窗口
LogRateLimitBurst=10000        # 窗口内最多 10000 条
```

### 6.2 实现（journald-server.c:1105-1121）

```c
// journald 在 server_dispatch_message() 中检查:
if (!journal_ratelimit_test(s->ratelimit,
                            c->unit,              // 按 unit 分桶
                            c->log_ratelimit_interval,
                            c->log_ratelimit_burst,
                            priority,
                            &s->last_ratelimit_drop_time)):
  // 超过限制，丢弃
  s->ratelimit_dropped++;
  return;
```

**特点**：
- 每个 unit 独立计数（不影响其他服务）
- 被限制时记录一条 "suppressed N messages" 汇总日志
- 默认值：interval=30s, burst=10000

## 9. LogNamespace=

### 9.1 配置

```ini
[Service]
LogNamespace=secure-logs
```

### 9.2 实现

```
效果:
  1. 服务的 stdout/stderr 连接到:
     /run/systemd/journal.secure-logs/stdout
     (而非默认的 /run/systemd/journal/stdout)

  2. 需要独立的 journald 实例:
     systemd-journald@secure-logs.service

  3. 日志存储在:
     /var/log/journal/<machine-id>.secure-logs/

  4. 查看:
     journalctl --namespace=secure-logs

用途:
  - 安全日志隔离（不同权限访问）
  - 高吞吐服务独立存储（不影响系统日志性能）
  - 不同保留策略
```

## 11. 完整日志数据流

```
┌───────────────┐
│  服务进程      │
│               │
│ printf("...")  │ ─── stdout ───→ ┌─────────────────────┐
│ fprintf(stderr)│ ─── stderr ───→ │  /run/systemd/       │
│ sd_journal_*  │ ─── native ───→ │  journal[.ns]/stdout │
│ syslog()      │ ─── syslog ───→ │  (/dev/log)          │
└───────────────┘                  └──────────┬──────────┘
                                              │
                                              ▼
                                   ┌─────────────────────┐
                                   │     journald        │
                                   │                     │
                                   │ 1. 识别 unit 来源    │
                                   │ 2. LogLevelMax 过滤  │
                                   │ 3. RateLimit 限流    │
                                   │ 4. 添加 ExtraFields  │
                                   │ 5. 添加元数据:       │
                                   │    _PID, _UID,      │
                                   │    _SYSTEMD_UNIT,   │
                                   │    _TRANSPORT,      │
                                   │    _BOOT_ID         │
                                   │ 6. 写入 journal 文件 │
                                   └──────────┬──────────┘
                                              │
                              ┌────────────────┼────────────────┐
                              ▼                ▼                ▼
                    /var/log/journal/   forward_to_syslog  forward_to_console
                    (持久存储)          (兼容 rsyslog)     (/dev/console)
```

## 13. 设计要点

### 13.1 结构化日志的优势

- 每条日志自动关联 `_SYSTEMD_UNIT`、`_PID`、`_UID` 等元数据
- 无需解析日志格式即可精确过滤
- `LogExtraFields=` 实现零代码的业务标签

### 13.2 零配置即可用

- 默认 `StandardOutput=journal`：服务无需任何日志配置即可获得结构化存储
- `SyslogLevelPrefix=true` 默认开启：应用通过简单的 `<N>` 前缀即可控制级别

### 13.3 命名空间隔离

`LogNamespace=` 实现了日志的物理隔离——不同命名空间有独立的 journald 实例、独立的存储、独立的访问控制。这对多租户和安全敏感场景至关重要。

### 13.4 性能保护

`LogRateLimitBurst=` 防止单个失控服务通过疯狂输出日志耗尽磁盘 I/O 和存储空间，同时不影响其他服务的日志记录。
